\title{Design, Implementation, and Evaluation of a Hybrid SQL/Graph Query Routing System}

\author{\IEEEauthorblockN{Priyanshu Mittal}
\IEEEauthorblockA{Final Project Documentation, April 2026}}

\maketitle

\begin{abstract}
This document provides the comprehensive engineering and algorithmic specification of a hybrid query routing system designed to evaluate execution environments (SQL vs. Graph representations) for mixed analytical workloads. The system processes a JSON-based Domain Specific Language (DSL), decomposes queries into dependency-aware topological graphs, extracts a 22-dimensional structural and historical feature vector, and delegates execution dynamically to either tabular (pandas/Spark SQL) or graph-based (GraphFrames) backends. We formalize the query routing problem, detail the dataset curation pipeline spanning LDBC SNB and OGB benchmarks, and rigorously evaluate rule-based versus learned routing policies. We outline the complete technology stack, including distributed deployment over Hadoop YARN and Spark. Empirical results on a strict, template-disjoint evaluation split reveal that while learned models achieve high classification performance ($F_1 = 0.9643$), they converge to match a simple, deterministic traversal heuristic, establishing a rigorous baseline for future work.
\end{abstract}

\section{Introduction and Problem Formulation}
Analytical queries increasingly intermix relational operations (e.g., joins, aggregations) with graph-topological traversals (e.g., multi-hop reachability). Static selection of a compute engine often yields sub-optimal latency. This project implements a dynamic routing framework that parses user queries, constructs an execution Plan, and models the routing decision as a classification task based on structural and historical features.

The implemented scope is engineering-complete for a HIFUN-inspired mixed-query subset and includes the following deliverables:

\item Dataset ingestion and curation pipeline for SNB, OGB, JOB/IMDB, and TPC-DS family sources.

\subsection{Formal Problem Definition}
Let $\mathcal{Q}$ be the set of all valid queries within our DSL. A query $q \in \mathcal{Q}$ undergoes decomposition into a directed acyclic graph (DAG) of algebraic operations, $G_q = (V, E)$, where $v \in V$ represents a subexpression block and edge $(u, v) \in E$ denotes that $v$ consumes the output of $u$.

The routing task is defined as a mapping function $R$:
\begin{equation}
R: V \to \{ \text{SQL}, \text{GRAPH} \}
\end{equation}
The optimal routing policy $R^*$ minimizes the total execution latency $L(G_q, R)$ plus the cross-engine data transfer penalty $\tau(u, v)$ for functionally adjacent nodes mapped to different target engines:
\begin{equation}
R^* = \arg\min_{R} \left( \sum_{v \in V} \text{cost}(v, R(v)) + \sum_{(u, v) \in E} \tau_{R(u) \to R(v)} \right)
\end{equation}
In this project, we cast this optimization as a supervised learning classification problem, using historical execution data to map $v$ to the statistically superior engine.

\subsection{AST Parsing and Validation}
The input query is expressed in a JSON-based DSL loosely inspired by HIFUN. The parsing layer validates the query structure utilizing strict `jsonschema` definitions. It implements unique operation ID checks, resolves parent-child relationships, and validates topological consistency by rejecting cyclic references.

\subsection{Query Decomposition}
The DAG decomposer collapses the validated AST into schedulable execution blocks. It identifies parallel execution boundaries, enabling independent subexpressions at the same dependency depth to be submitted to the routing engine concurrently.

\subsection{Routing Runtime Semantics}
Routing occurs at the subexpression level. For a topological layer $\ell$, all nodes with in-degree zero after prior layer completion are eligible for concurrent dispatch. Let $\mathcal{V}_\ell$ denote this layer. The runtime stage cost is:
\begin{equation}
T_\ell = \max_{v \in \mathcal{V}_\ell} t(v, R(v)) + \sum_{(u,v) \in E_\ell} \tau_{R(u) \to R(v)}
\end{equation}
and total execution time is approximated by $\sum_{\ell} T_\ell$. This formulation explains why transfer penalties can dominate even when per-node engine cost is low.

\section{Execution Infrastructure and Shims}

\subsection{Technology Stack and Distributed Provisioning}
The execution environment is strictly containerized utilizing Docker Compose to ensure reproducible deployment environments:
\begin{itemize}[leftmargin=*]
\item \textbf{Storage and Resource Limits:} Hadoop HDFS (NameNode, DataNode) scaling to distribute parquet artifacts, and Apache YARN for cluster resource negotiation.
\item \textbf{Compute:} PySpark 3.4.2 Standalone (Master + Multiple Workers), configured with the Spark History Server.
\item \textbf{Scientific Tooling:} JupyterLab environment bridging PySpark with XGBoost, LightGBM, and scikit-learn for off-line training phases.
\end{itemize}

\subsection{Execution Pathways}
The final mapped subexpressions propagate into backend shims:
\begin{enumerate}
    \item \textbf{SQL Pathway:} Leveraging `SparkSQLGenerator` and `pandas_sql_generator` (for rapid local staging). It translates projection, filter, and join predicates directly into relational projections.
    \item \textbf{GRAPH Pathway:} Managed by the `GraphFramesGenerator`. Operations featuring recursive expansions map perfectly via Motif Finding and PageRank APIs defined in GraphFrames DSL.
\end{enumerate}

Dependency synchronization tracks output manifestations. If $R(u)=\text{GRAPH}$ and $R(v)=\text{SQL}$, the system injects a materialization step to flatten the GraphFrame vertex and edge attributes back into a relational DataFrame before injecting it into $v$.

\section{Dataset Engineering and Curation}

To facilitate rigorous evaluation, the system provisions real-world graph and tabular datasets.

\subsection{Ingestion Pipelines}
Scripts were written to synthesize datasets into high-density Apache Parquet files for uniform access latency:
\begin{itemize}[leftmargin=*]
\item \textbf{LDBC SNB:} Social network schemas, heavily relying on bidirectional user-knows-user relationships.
\item \textbf{Open Graph Benchmark (OGB):} Citations topologies converting OGB raw formats to typed graph datasets.
\item \textbf{Classic Relational:} TPC-DS and JOB/IMDB tabular references to measure edge-case non-graph queries.
\end{itemize}

\subsection{Strict Data Separation Mechanism}
A major architectural contribution is strict dataset split curation. Standard randomized cross-validation permits data leakage because isomorphic query structures duplicate across sets. We enforced a \textbf{template-disjoint split}:
\begin{equation}
\forall q_{tr} \in \mathcal{D}_{train}, \forall q_{te} \in \mathcal{D}_{test} : T(q_{tr}) \cap T(q_{te}) = \emptyset
\end{equation}
Under this rigorous constraint, the curated evaluation set consisted of $N_{train}=230$ independent training cases and $N_{eval}=58$ strict evaluation templates.

\subsection{Data Quality Gates}
Before model training or reporting, each dataset candidate is validated against three gates:
\begin{itemize}[leftmargin=*]
\item \textbf{Coverage gate:} verifies minimum per-dataset support for both SQL and GRAPH labels.
\item \textbf{Runtime gate:} filters rows with invalid, timed-out, or inconsistent backend runtime measurements.
\item \textbf{Template gate:} confirms no template overlap between fixed train and fixed evaluation splits.
\end{itemize}
Only artifacts passing all gates are admitted into strict result tables.

\subsection{Labeling Methodology}
Ground-truth routing labels are assigned from measured backend runtimes. For each routable unit $v$, both candidate backends are executed in controlled conditions and labeled by:
\begin{equation}
y_v = \arg\min_{e \in \{\text{SQL},\text{GRAPH}\}} t(v,e)
\end{equation}
Rows with failed execution for both engines are discarded; rows with one successful engine are retained when diagnostics indicate deterministic failure in the other path.

\section{Experimental Rigor and Empirical Results}

\subsection{Experiment Protocol}
All reported strict metrics follow the same protocol:
\begin{enumerate}[leftmargin=*]
\item Train using only fixed strict training rows.
\item Tune minimal hyperparameters on training-only internal folds.
\item Freeze model and evaluate once on fixed strict evaluation rows.
\item Report point metrics and bootstrap confidence intervals.
\item Compare against deterministic baselines under identical labels.
\end{enumerate}
This protocol avoids adaptive overfitting on the final evaluation split.

\subsection{Evaluation Formalization}
Let $TP$, $TN$, $FP$, and $FN$ represent the standard classification matrices focusing on GRAPH classification superiority. We optimize for high $F_1$-score while plotting Precision-Recall Area Under Curve (PR-AUC) metrics to map robustness across shifting probability configurations:
\begin{equation*}
F_1 = 2 \cdot \left( \frac{\text{Precision} \cdot \text{Recall}}{\text{Precision} + \text{Recall}} \right)
\end{equation*}

\subsection{Robustness and Ablation Observations}
The tie between XGBoost and TraversalRule reveals critical insights about the problem space. We conducted an ablation study to mask specific feature tensors (e.g., removing historical $\mu_{time}$ inputs via occlusion masks $M_{hist} \odot \mathbf{x}$). Ablation showed $\Delta F_1 < 0.01$ degradation. Because the current curated dataset predominantly maps structural topology directly to backend affinities, advanced non-linear estimators do not dramatically exceed standard heuristics.
We computed 10,000 bootstrap iteration intervals, confirming with 95\% confidence that ML methodologies operate equally to hand-crafted routing logic for these schemas.

\subsection{Interpretation and Statistical Caution}
The strict result should be interpreted as \textbf{parity evidence}, not superiority evidence. With a small evaluation sample, uncertainty intervals remain wide near the upper bound. Consequently, the scientifically correct statement is that learned policies currently reproduce the heuristic boundary already captured by traversal-centric signals.

Correctness is enforced at multiple levels:
\begin{itemize}[leftmargin=*]

\item \textbf{Execution tests:} compare generated SQL/graph plans against expected intermediate outputs.

\section{Threats to Validity}
\begin{itemize}[leftmargin=*]
\item \textbf{Sample size risk:} strict evaluation size ($N_{eval}=58$) limits sensitivity for small performance differences.
\item \textbf{Dataset concentration risk:} strict family is dominated by SNB-like and OGB-like templates.
\item \textbf{Label noise risk:} runtime-based labels can be affected by transient cluster conditions.
\item \textbf{Shift inconsistency risk:} cross-family transfer can collapse when class support is imbalanced.
\item \textbf{Backend asymmetry risk:} failure behavior may differ between SQL and graph engines for edge patterns.
\end{itemize}

\section{Engineering Decisions and Trade-offs}
Key implementation decisions and their trade-offs are summarized below:
\begin{itemize}[leftmargin=*]
\item \textbf{Rule plus ML coexistence:} keeps a reliable fallback baseline but can reduce perceived novelty.
\item \textbf{Feature-based models instead of deep encoders:} faster iteration and interpretability, but weaker transfer capacity.
\item \textbf{Strict split prioritization:} improves claim integrity while reducing sample volume.
\item \textbf{Dual backend support:} enables realistic mixed execution but introduces materialization overhead.
\item \textbf{Containerized stack:} reproducible deployment at the expense of operational complexity.
\end{itemize}

\section{Lessons Learned}
\begin{enumerate}[leftmargin=*]
\item Reproducibility infrastructure must be developed in parallel with modeling, not after headline results.
\item Baseline heuristics should be treated as primary competitors from day one.
\item Template-level leakage prevention is mandatory for routing tasks with structural repetition.
\item Robustness and shift tests are essential to prevent over-claiming from in-domain performance.
\item Honest parity reporting is scientifically stronger than unsupported superiority claims.
\end{enumerate}

\section{Conclusion and Future Work}
\textbf{Conclusion:} We successfully executed a large-scale integration of a DSL parser, a unified execution DAG decomposer, robust PySpark container orchestration, and a 22-dimensional feature engineering routing plane. Empirical findings dictate absolute modesty: state-of-the-art tree-boosting classifiers successfully identify patterns, but they ultimately synthesize behaviors matching syntactic structural heuristics like the TraversalRule. The project provides a meticulously scalable bedrock for continued hybrid database research and a defensible evaluation framework for future iterations.

\textbf{Future Directions:} Immediate research must target deep representation-learning matrices for AST blocks (Graph Neural Networks instead of scalar vectors), expanded real-world tabular data integration from decoupled domains, and optimizations minimizing latency $L(G_q)$ by anticipating Spark cluster broadcast overhead operations dynamically across node boundaries.
