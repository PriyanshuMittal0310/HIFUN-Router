\title{Final Project Report: Honest Evaluation of a Learned SQL/Graph Router}

\author{\IEEEauthorblockN{Priyanshu Mittal}
\IEEEauthorblockA{Project Report, April 2026}}

\maketitle

\begin{abstract}
This report presents the final, evidence-aligned state of a hybrid query routing prototype that chooses between SQL and graph execution for decomposed subexpressions. The original objective was to test whether a learned router outperforms simple rule-based routing on mixed SQL/graph workloads. The final result is more modest: on the strict held-out split currently available, learned models (Logistic Regression, Decision Tree, XGBoost) \emph{tie} a one-line traversal heuristic at F1 = 0.9643. This report therefore avoids superiority claims and documents what is supported by data: (1) a working parser--decomposer--router--executor pipeline, (2) reproducible strict evaluation artifacts on real measurements, and (3) clear limitations around dataset coverage, transfer stability, and unresolved correctness issues in selected mixed queries. All numbers shown are taken from current repository artifacts or recomputed from current strict split files.
\end{abstract}

\begin{IEEEkeywords}
Query Routing, Hybrid Query Processing, Graph Analytics, Spark SQL, GraphFrames, Reproducibility
\end{IEEEkeywords}

\section{Problem Statement (Final and Realistic)}
Modern analytics frequently mix relational and graph operators. A practical systems question is whether query subexpressions should execute on SQL engines (e.g., Spark SQL) or graph engines (e.g., GraphFrames). This project implements a routing pipeline and evaluates:
\begin{enumerate}
\item Can learned routing \emph{at least match} simple heuristics on available real labels?
\item Is there evidence of robust cross-dataset generalization?
\item Is the implementation reproducible and honest about limitations?
\end{enumerate}

The implemented language is a HIFUN-inspired JSON DSL subset (FILTER, MAP, JOIN, TRAVERSAL, AGGREGATE), not full formal HIFUN semantics.

\section{System Summary}
The implemented pipeline has five stages: DSL parsing, dependency-aware decomposition, 22-feature extraction, routing prediction, and engine-specific execution with result composition. The router supports both a fast pandas test path and a Spark/GraphFrames path.

\section{Method and Reproducibility}
We evaluate deterministic baselines and three learned classifiers on the fixed strict split:
\begin{itemize}
\item AlwaysSQL baseline
\item TraversalRule baseline: if \texttt{has\_traversal}=1 then GRAPH else SQL
\item Logistic Regression (balanced)
\item Decision Tree (balanced)
\item XGBoost (200 trees, depth 5, learning rate 0.05)
\end{itemize}

Features are the 22 engineered routing features from the project pipeline. Metrics are computed on the held-out strict eval split.

\textbf{Reproducibility:} this repository now includes four notebook scaffolds:
\texttt{notebooks/01\_data\_exploration.ipynb},
\texttt{notebooks/02\_feature\_analysis.ipynb},
\texttt{notebooks/03\_model\_training.ipynb},
\texttt{notebooks/04\_results\_visualization.ipynb}.

\textbf{Uncertainty.} Bootstrap on strict eval for XGBoost gives F1 95\% CI \([0.9057, 1.0000]\), median 0.9667 (2000 resamples). With only 58 eval rows, small changes can move point estimates.

\section{What This Project Demonstrates vs. Does Not}
\textbf{Supported by evidence:}
\begin{itemize}
\item End-to-end hybrid routing system is implemented and testable.
\item Learned routing can match simple heuristic quality on current strict split.
\item Strict artifacts and notebook scaffolds improve reproducibility.
\end{itemize}

\textbf{Not supported (and therefore not claimed):}
\begin{itemize}
\item Learned routing superiority over a simple traversal rule.
\item Robust cross-dataset generalization across all workload families.
\item End-to-end latency improvement target (e.g., +10\% median) over rule baseline.
\end{itemize}

\section{Threats to Validity}
\begin{itemize}
\item \textbf{Small eval sample:} strict eval has 58 rows.
\item \textbf{Narrow dataset coverage:} strict curated data is SNB/OGB only.
\item \textbf{Transfer instability:} broader dataset-shift artifacts include zero-F1 regimes and single-class-train failures.
\item \textbf{Implementation confounders:} some mixed-query correctness issues remain around multi-hop traversal schema normalization.
\end{itemize}

\section{Conclusion}
This final report positions the work as a solid systems prototype with transparent evaluation rather than a finished routing-optimization breakthrough. The honest result is a tie between learned models and a simple heuristic on the current strict split. The most valuable next milestone is a larger, truly mixed real-measurement dataset and controlled end-to-end latency experiments against AlwaysSQL, AlwaysGraph, and rule-based routing.
